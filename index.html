BEFORE

Pain Points

PM 文件工作的六大重複痛：寫得慢、review 沒紀錄、規格漂移、發布鏈手動接

📄

PRD 寫得慢

散亂的 Google Docs、Slack threads、PDF 需手動整理成結構化 PRD，每次 2-4 小時

📜

沒有 Product Spec SSOT

系統現況散落在 codebase 與工程師腦袋裡。新 PRD 跟現況衝突，無法自動偵測

👥

Review 缺結構

PM/Eng/QA/PD 四角度 review 用 Google Doc 評論散著，找不到 must-fix 和 nice-to-have 的差別

🧪

測試追蹤困難

Test cases 散落 PRD 各處，無法自動產 Playwright 腳本或追每次測試結果

📢

Release Note 重工

每次從 PRD 手動摘要，重新排版 Internal Update，手動貼 Slack

🌐

翻譯流程瑣碎

EN/TH/JA 三語翻譯手動對照術語表，review request 手動發 Slack，發布手動跑 CLI

SOLUTION

What We Built

16 個 Claude Code Skills，分三層覆蓋從探索到上線的完整 PM 工作流

1
探索層 — /product-thinking：ICE 評分 / 四大風險掃描 / 成功指標 / 假設句型，引導 PM 結構化評估功能需求
2
PRD Lifecycle（Git-as-SSOT） — /new-prd, /sync-prd, /review-prd（4 subagent 平行 PM/Eng/QA/PD review）, /sync-gdoc, /pull-gdoc-comments, /archive-prd, /gen-product-spec。PRD 與 Spec 住在 Git，Google Doc 是 read-only snapshot
3
上線發布鏈 — /gen-test-cases → /test-run（--auto Playwright）→ /gen-release-notes（推 Slack）→ /translate（cl-locales）→ /verify。/release-pipeline 一鍵串接，/ga-tracking 處理 GA4 SPA 埋碼，/gen-hc-content 產 Help Center 素材

完整 16 skill 表 & 目錄結構請見 README。

QUICK START

How to Start

五步驟開始使用

1

Clone GitHub Repo

git clone https://github.com/Lydia47/pm-release-accelerator.git

2

安裝 Skills 到 Claude Code

cd pm-release-accelerator && cp -r .claude/skills/* ~/.claude/skills/

3

安裝 & 認證 gws CLI

npm install -g @googleworkspace/cli
gws auth setup → gws auth login -s drive,sheets,docs

4

開始使用

跟 AI 對話探索問題 → 需要時 /product-thinking 評估 → /new-prd 建立 PRD draft，後續可 /sync-prd、/review-prd

5

下游串接

PRD 定稿後 → /release-pipeline <prd-name> 一鍵串 test → release notes → translate → sync-gdoc
